Atue como especialista em SQL Server, modelagem de banco de dados e scripts de
migração estrutural seguros.

Vou anexar o script CREATE atual da tabela [<<<NOME_DA_TABELA>>>].
Sua tarefa é gerar um script SQL Server completo e blindado que recrie essa
tabela com os novos campos abaixo, migre 100% dos dados e substitua a tabela
original, preservando a integridade do banco.

== NOVOS CAMPOS (definição exata e posição exata) ==
Para cada campo, respeite rigorosamente: nome, tipo, tamanho, nulabilidade,
default e a posição informada. "Após o campo X" significa imediatamente após X.

<<<EXEMPLO — substituir pelos campos reais:
1. [movimento_devolvido]  [bit]          NOT NULL  DEFAULT ((0))   — após [obs_cancelamento]
2. [id_usuario_devolucao] [int]          NOT NULL  DEFAULT ((0))   — após [movimento_devolvido]
3. [datahora_devolucao]   [datetime2](0) NULL      DEFAULT (NULL)  — após [id_usuario_devolucao]
4. [obs_devolucao]        [varchar](max) NULL      DEFAULT (NULL)  — após [datahora_devolucao]
>>>

Valor inicial dos novos campos para as linhas já existentes na migração:
<<<ex.: movimento_devolvido = 0; id_usuario_devolucao = 0; demais = NULL>>>
Se algo na definição ou posição for ambíguo, pergunte antes de gerar o script.

== ESTRATÉGIA OBRIGATÓRIA ==
1. Tabela auxiliar com a estrutura original completa + novos campos, criada SEM
   constraints nomeadas conflitantes (PK com nome temporário; defaults criados
   só após o swap, com os nomes canônicos).
2. Capturar DINAMICAMENTE do catálogo do banco (não apenas do script-fonte,
   que pode estar defasado): constraints DEFAULT (nome/coluna/definição),
   índices não clusterizados e UNIQUE constraints (key, INCLUDE, filtro),
   e FOREIGN KEYS nas duas direções (de e para a tabela), com regras
   ON DELETE/ON UPDATE e estado trusted — recriando tudo após o swap.
3. Cópia dos dados com SET IDENTITY_INSERT, lista de colunas montada
   dinamicamente pela interseção das duas tabelas, e reseed explícito do
   IDENTITY (DBCC CHECKIDENT) com base no maior id migrado.
4. Cópia + validação de contagem + drop da original + rename + recriação de
   constraints numa ÚNICA transação com TABLOCKX na origem (swap atômico:
   qualquer falha => ROLLBACK total, tabela original intacta).

== PRÉ-VALIDAÇÕES OBRIGATÓRIAS (abortar com mensagem clara) ==
- Tabela original inexistente.
- Idempotência: todos os novos campos já existem => abortar como "já aplicado";
  apenas parte deles existe => abortar como "estado parcial, análise manual".
- Triggers na tabela; objetos com SCHEMABINDING referenciando a tabela;
  CDC habilitado; replicação; tabela temporal.
- Avisos não fatais (PRINT) para permissões GRANT/DENY explícitas e extended
  properties, que se perdem no DROP e devem ser reaplicadas pelo DBA.

== BLINDAGEM E IDEMPOTÊNCIA ==
- Limpeza inicial de resíduos de execuções anteriores (tabela auxiliar,
  constraints órfãs com nomes temporários, FKs apontando para a auxiliar).
- Limpeza GLOBAL POR NOME antes de criar qualquer constraint: se o nome existir
  preso a qualquer objeto, remover do dono atual antes de criar.
- IF EXISTS / IF NOT EXISTS em toda criação, alteração, drop e rename.
- Cada bloco autocontido com TRY/CATCH local; falha fatal => THROW interrompendo
  tudo; CATCH do bloco crítico faz ROLLBACK e remove a tabela auxiliar.
- Limpeza final de todos os objetos de trabalho (tabelas temporárias etc.).
- Verificação final: contagem de linhas, existência de todas as novas colunas,
  IDENT_CURRENT, contagem de defaults e FKs, ausência de resíduos.

== QUALIDADE TÉCNICA ==
- Validar os nomes de colunas das views de catálogo (sys.*) contra a
  documentação oficial antes de usar (ex.: sys.sql_modules.is_schema_bound) —
  não escrever consultas de catálogo de memória.
- Compatibilidade mínima SQL Server 2012+ (usar FOR XML PATH em vez de
  STRING_AGG; omitir opções exclusivas de versões novas, comentando o motivo,
  como OPTIMIZE_FOR_SEQUENTIAL_KEY quando o valor original é OFF).
- Organizar em blocos numerados e comentados: pré-validações; limpeza inicial;
  criação da auxiliar; captura dinâmica de objetos; cópia + swap atômico;
  recriação de defaults; recriação de índices; recriação de FKs; verificação
  final + limpeza.
- Marcar com comentário ">>> NOVOS CAMPOS <<<" todos os pontos que precisam ser
  editados ao adicionar campos futuros.
