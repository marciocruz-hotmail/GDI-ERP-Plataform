# 4. Critérios de avaliação

## 4.1 Tecnologias obsoletas ou com risco

Identifique:

- Uso de versões antigas de bibliotecas NuGet.
- Uso de bibliotecas JavaScript antigas.
- Uso de APIs depreciadas do .NET Framework.
- Código dependente de recursos antigos do ASP.NET MVC.
- Dependências sem manutenção evidente.
- Uso de bibliotecas duplicadas para a mesma finalidade.
- Mistura de Bootstrap antigo com Bootstrap 5.
- Uso de jQuery ou plugins legados onde houver risco técnico.
- Uso de scripts inline em excesso nos .cshtml.
- Uso de Web.config com configurações sensíveis ou frágeis.
- Uso de pacotes que possam ter vulnerabilidades conhecidas, quando identificável pelo nome e versão.

Para cada item encontrado, informe:
- Tecnologia ou componente.
- Arquivo onde foi identificado.
- Evidência encontrada.
- Risco associado.
- Gravidade.
- Recomendação.

## 4.2 Acesso a dados

Avalie se o projeto mistura Entity Framework e ADO.NET de forma inadequada.

Identifique:

- Fluxos onde Controller usa EF diretamente.
- Fluxos onde Controller usa ADO.NET diretamente.
- Fluxos onde Repository usa EF e ADO.NET misturados.
- Queries SQL montadas por concatenação de string.
- Queries com risco de SQL Injection.
- Falta de parametrização.
- Uso excessivo de DataTable, DataSet, SqlDataReader ou SqlCommand diretamente em camadas inadequadas.
- Falta de separação entre regra de negócio e acesso a dados.
- Uso inconsistente de transações.
- Consultas repetidas ou duplicadas.
- Consultas potencialmente pesadas para telas do ERP.
- Ausência de paginação em consultas usadas por DataTables.

Classifique cada ocorrência por:
- Fluxo funcional.
- Arquivos envolvidos.
- Padrão usado.
- Problema identificado.
- Risco.
- Recomendação.

## 4.3 Arquivos mortos

Identifique arquivos aparentemente não utilizados pelo ERP.

Considere como possível arquivo morto:

- Arquivo .cs sem referência em outros arquivos.
- Controller sem rota, sem referência ou sem views associadas.
- View .cshtml sem Action correspondente.
- ViewModel não referenciado.
- Model não referenciado.
- Helper não referenciado.
- Service não referenciado.
- Repository não referenciado.
- Arquivo .js não incluído em BundleConfig, Layout, View ou referência direta.
- Arquivo .css não incluído em BundleConfig, Layout ou View.
- Imagem, plugin ou asset não referenciado.
- Arquivo duplicado ou versão antiga mantida no projeto.
- Arquivo de backup, teste, cópia ou temporário dentro do projeto.

Para cada arquivo suspeito, classifique como:

- Morto provável: não há referência encontrada.
- Morto possível: há indícios de não uso, mas precisa de confirmação manual.
- Legado preservado: parece antigo, mas pode estar vinculado a fluxo específico.
- Não classificado: evidência insuficiente.

Não recomende exclusão automática.
Recomende validação manual antes de remover qualquer arquivo.

Para cada arquivo suspeito, informe:
- Caminho completo do arquivo.
- Tipo do arquivo.
- Evidência de não uso.
- Impacto potencial se removido indevidamente.
- Nível de confiança.
- Recomendação.

# 5. Classificação de riscos

Classifique cada risco usando a matriz abaixo.

Gravidade:
- Crítica: pode causar falha grave, indisponibilidade, vazamento de dados, perda de dados ou bloqueio operacional.
- Alta: pode causar erro relevante, lentidão importante, vulnerabilidade ou alto custo de manutenção.
- Média: prejudica evolução, manutenção ou experiência do usuário, mas sem impacto imediato grave.
- Baixa: melhoria técnica desejável, organização ou redução de dívida técnica.

Probabilidade:
- Alta: forte evidência no código ou uso frequente.
- Média: evidência parcial ou depende do fluxo de uso.
- Baixa: possibilidade técnica, mas sem evidência de uso frequente.

Impacto:
- Operacional.
- Segurança.
- Performance.
- Manutenção.
- Evolução.
- Usabilidade.
- Publicação.
- Banco de dados.

Prioridade:
- P0: corrigir com urgência.
- P1: corrigir no curto prazo.
- P2: planejar correção.
- P3: melhoria futura.
- P4: apenas monitorar ou validar.
